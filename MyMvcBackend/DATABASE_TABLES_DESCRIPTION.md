# Описание таблиц базы данных

## 📊 Общее количество таблиц: 10

---

## Существующие таблицы (5)

### 1. **Users** - Пользователи
**Назначение:** Хранит информацию о зарегистрированных пользователях системы.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `Name` - Имя пользователя
- `Email` - Email адрес (уникальный)
- `PasswordHash` - Хеш пароля
- `EmailVerified` - Статус верификации email
- `VerificationToken` - Токен для верификации
- `CreatedAt`, `UpdatedAt` - Даты создания и обновления

**Связи:**
- Связана с `Seats` (резервация мест)
- Связана с `UserTickets` (билеты пользователя)
- Связана с `Orders` (заказы пользователя)
- Связана с `Payments` (платежи)
- Связана с `Reviews` (отзывы)
- Связана с `Notifications` (уведомления)

---

### 2. **Events** - События
**Назначение:** Хранит информацию о всех событиях/мероприятиях.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `Title` - Название события
- `Image` - Путь к изображению
- `Date`, `Time` - Дата и время события
- `Location`, `Address` - Место проведения
- `Price` - Цена (строка, может содержать диапазон)
- `Category` - Категория события
- `Description` - Описание
- `EventType` - Тип события
- `Lineup` - Состав исполнителей (JSON)
- `IsFeatured` - Признак избранного события

**Связи:**
- Связана с `TicketTypes` (типы билетов)
- Связана с `Seats` (места в зале)
- Связана с `UserTickets` (проданные билеты)
- Связана с `Reviews` (отзывы о событии)
- Связана с `Notifications` (уведомления о событии)

---

### 3. **TicketTypes** - Типы билетов
**Назначение:** Определяет различные типы билетов для каждого события (VIP, Стандарт, Партер и т.д.).

**Основные поля:**
- `Id` - Уникальный идентификатор
- `Name` - Название типа билета
- `Price` - Цена билета
- `Available` - Доступность типа билета
- `EventId` - Связь с событием

**Связи:**
- Связана с `Events` (принадлежит событию)

**Польза:** Позволяет создавать разные категории билетов с разными ценами для одного события.

---

### 4. **Seats** - Места
**Назначение:** Хранит информацию о каждом месте в зале для каждого события.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `EventId` - Связь с событием
- `Row` - Ряд
- `Number` - Номер места
- `Status` - Статус (available, reserved, sold)
- `Type` - Тип места (standard, vip, disabled)
- `Price` - Цена места
- `ReservedByUserId` - Пользователь, который зарезервировал место
- `ReservationExpiresAt` - Время истечения резервации

**Связи:**
- Связана с `Events` (принадлежит событию)
- Связана с `Users` (резервация пользователем)
- Связана с `UserTickets` (проданное место)

**Польза:** Позволяет реализовать выбор конкретных мест на карте зала, резервацию мест на время.

---

### 5. **UserTickets** - Билеты пользователей
**Назначение:** Хранит информацию о купленных билетах пользователей.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `UserId` - Пользователь-владелец билета
- `EventId` - Событие
- `SeatId` - Место
- `TicketType` - Тип билета
- `Price` - Цена покупки
- `PurchaseDate` - Дата покупки
- `EventDate` - Дата события
- `QrCode` - QR-код билета
- `IsUsed` - Статус использования билета

**Связи:**
- Связана с `Users` (владелец билета)
- Связана с `Events` (событие)
- Связана с `Seats` (место)
- Связана с `Notifications` (уведомления о билете)

**Польза:** Централизованное хранение всех билетов пользователей, возможность проверки QR-кодов, отслеживание использования.

---

## Новые таблицы (5)

### 6. **Orders** - Заказы
**Назначение:** Группирует несколько билетов в один заказ. Позволяет отслеживать полную стоимость покупки и статус заказа.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `UserId` - Пользователь, сделавший заказ
- `OrderNumber` - Уникальный номер заказа (для отслеживания)
- `TotalAmount` - Общая сумма заказа
- `Status` - Статус заказа (pending, paid, cancelled, refunded)
- `CreatedAt` - Дата создания заказа
- `CompletedAt` - Дата завершения заказа

**Связи:**
- Связана с `Users` (пользователь)
- Связана с `Payments` (платежи по заказу)

**Польза:**
- ✅ Позволяет группировать несколько билетов в одну покупку
- ✅ Упрощает отслеживание статуса оплаты
- ✅ Уникальный номер заказа для клиентской поддержки
- ✅ История всех заказов пользователя

**Пример использования:** Пользователь покупает 3 билета на концерт - все они объединяются в один заказ с общей суммой.

---

### 7. **Payments** - Платежи
**Назначение:** Хранит информацию о всех платежах. Отслеживает способ оплаты, статус транзакций, историю платежей.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `OrderId` - Связь с заказом
- `UserId` - Пользователь, совершивший платеж
- `Amount` - Сумма платежа
- `PaymentMethod` - Способ оплаты (card, bank_transfer, e_wallet)
- `Status` - Статус платежа (pending, completed, failed, refunded)
- `TransactionId` - ID транзакции от платежной системы
- `PaymentDetails` - Дополнительная информация (JSON)
- `CreatedAt` - Дата создания платежа
- `CompletedAt` - Дата завершения платежа

**Связи:**
- Связана с `Orders` (платеж по заказу)
- Связана с `Users` (пользователь)

**Польза:**
- ✅ Полная история всех платежей
- ✅ Интеграция с платежными системами (через TransactionId)
- ✅ Отслеживание неудачных платежей
- ✅ Возможность возврата средств (refunded)
- ✅ Разные способы оплаты
- ✅ Аудит финансовых операций

**Пример использования:** Пользователь оплачивает заказ картой - создается запись в Payments с TransactionId от банка.

---

### 8. **Reviews** - Отзывы
**Назначение:** Позволяет пользователям оставлять отзывы и оценки событиям. Помогает другим пользователям принимать решения.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `UserId` - Пользователь, оставивший отзыв
- `EventId` - Событие, на которое оставлен отзыв
- `Rating` - Оценка от 1 до 5
- `Comment` - Текстовый отзыв
- `CreatedAt` - Дата создания отзыва
- `UpdatedAt` - Дата обновления отзыва
- `IsApproved` - Статус модерации (для фильтрации нецензурных отзывов)

**Связи:**
- Связана с `Users` (автор отзыва)
- Связана с `Events` (событие)

**Ограничения:**
- Один пользователь может оставить только один отзыв на событие (уникальный индекс)

**Польза:**
- ✅ Повышает доверие к платформе
- ✅ Помогает пользователям выбирать события
- ✅ Модерация отзывов (IsApproved)
- ✅ Расчет средней оценки события
- ✅ Улучшение качества сервиса на основе отзывов

**Пример использования:** После посещения концерта пользователь оставляет отзыв с оценкой 5 и комментарием "Отличное мероприятие!".

---

### 9. **Notifications** - Уведомления
**Назначение:** Хранит все уведомления для пользователей. Позволяет отправлять уведомления о новых событиях, изменениях в билетах, напоминаниях.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `UserId` - Пользователь, которому отправлено уведомление
- `Title` - Заголовок уведомления
- `Message` - Текст уведомления
- `Type` - Тип уведомления (info, warning, success, error, reminder)
- `IsRead` - Статус прочтения
- `CreatedAt` - Дата создания
- `ReadAt` - Дата прочтения
- `RelatedEventId` - Связь с событием (опционально)
- `RelatedTicketId` - Связь с билетом (опционально)

**Связи:**
- Связана с `Users` (получатель уведомления)
- Связана с `Events` (если уведомление связано с событием)
- Связана с `UserTickets` (если уведомление связано с билетом)

**Польза:**
- ✅ Уведомления о новых событиях в интересующих категориях
- ✅ Напоминания о предстоящих событиях
- ✅ Уведомления об изменениях в билетах
- ✅ Уведомления о статусе платежей
- ✅ Централизованная система уведомлений
- ✅ История всех уведомлений пользователя

**Пример использования:** 
- "Напоминание: Ваше событие через 2 дня!"
- "Ваш платеж успешно обработан"
- "Новое событие в категории 'Концерты'"

---

### 10. **ContactMessages** - Сообщения контактов
**Назначение:** Сохраняет все обращения пользователей через форму обратной связи. Позволяет отслеживать обращения, отвечать на них и анализировать популярные вопросы.

**Основные поля:**
- `Id` - Уникальный идентификатор
- `Name` - Имя отправителя
- `Email` - Email отправителя
- `Message` - Текст сообщения
- `Status` - Статус обработки (new, in_progress, resolved, archived)
- `CreatedAt` - Дата получения сообщения
- `ResolvedAt` - Дата решения вопроса
- `Response` - Ответ администратора
- `RespondedByUserId` - ID администратора, который ответил

**Связи:**
- Нет прямых связей (необязательно, можно добавить связь с Users если пользователь зарегистрирован)

**Польза:**
- ✅ Сохранение всех обращений пользователей
- ✅ Отслеживание статуса обработки обращений
- ✅ История ответов администраторов
- ✅ Анализ популярных вопросов
- ✅ Улучшение качества поддержки
- ✅ Не теряются обращения (в отличие от только email-уведомлений)

**Пример использования:** Пользователь отправляет вопрос через форму обратной связи - создается запись в ContactMessages со статусом "new". Администратор отвечает - статус меняется на "resolved", заполняется поле Response.
